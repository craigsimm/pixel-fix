## Verified bugs

### 1) Major - test suite is not portable to Tk-less Linux environments

- **Severity:** Major
- **Failing tests:**
  - `tests/test_gui_builtin_palettes.py` (module import failure)
  - `tests/test_gui_processing.py` (module import failure)
- **Failure detail:** `python3 -m pytest` aborts during collection with `ModuleNotFoundError: No module named 'tkinter'`.

#### Root cause analysis

The GUI dependency is imported at module import time in multiple places:

- `tests/test_gui_builtin_palettes.py:4` imports `tkinter as tk`
- `tests/test_gui_processing.py:5` imports `pixel_fix.gui.app`
- `src/pixel_fix/gui/app.py:5` imports `tkinter as tk`

`tests/test_gui_builtin_palettes.py` contains a runtime skip in `_build_gui()`, but that skip is never reached when `tkinter` is unavailable because collection already fails at the top-level import. `tests/test_gui_processing.py` is also blocked even though most of its tests exercise non-window logic via `PixelFixGui.__new__`, because importing `pixel_fix.gui.app` immediately imports Tk.

This makes the repository's default test command fail on fresh/headless Linux environments unless the system package providing Tk is preinstalled.

#### Steps to reproduce

1. Start from a Linux environment without the Tk system package installed.
2. Install project/test dependencies:
   ```bash
   python3 -m pip install -e . pytest
   ```
3. Run:
   ```bash
   python3 -m pytest
   ```
4. Observe collection stop with `ModuleNotFoundError: No module named 'tkinter'`.

#### User impact

- CI or automation jobs on minimal Linux images fail before executing the actual test suite.
- Contributors following the current README development install instructions can hit the same failure locally.
- Non-GUI regressions are masked because collection stops early.

#### Recommended fixes

1. Guard GUI-only tests with `pytest.importorskip("tkinter")` or an equivalent module-level skip.
2. Split non-UI logic from `src/pixel_fix/gui/app.py` so tests that do not need a Tk runtime can import lightweight helpers without importing the full window stack.
3. Add a CI strategy decision:
   - either install the Tk system package in jobs that collect GUI tests, or
   - explicitly exclude/mark GUI tests in headless jobs.
4. Update Linux setup docs to mention the required Tk package.

## Improvement opportunities

### A) Performance bottleneck in `clean_orphan_pixels(..., fill_gaps=True)`

- **Area:** `src/pixel_fix/cleanup/filters.py`
- **Severity:** Minor
- **Observation:** The gap-fill path performs repeated Python-level full-grid scans per distinct label and repeatedly recomputes neighbor counts. On this runner, a quick benchmark of a synthetic `256x256` image with 8 colors took about **2.88s** for `clean_orphan_pixels(..., fill_gaps=True)`, versus about **0.03s** for `remove_anti_aliased_edges(...)` on the same image size.
- **Why it matters:** The new cleanup controls are in the interactive downsample path, so large inputs could make the GUI feel sluggish when gap filling is enabled.
- **Suggestions:**
  - cache local neighbor counts instead of recalculating them repeatedly;
  - short-circuit the per-label pass when support cannot reach `_MIN_GAP_CLAIM_NEIGHBORS`;
  - consider vectorized morphology/counting for the fill-gaps path.

### B) Missing coverage for Tk-less environments

- **Area:** test strategy / CI
- **Severity:** Minor
- **Observation:** The suite covers GUI behavior, but it does not protect the repository against collection failures when Tk is unavailable.
- **Why it matters:** The current failure is a portability issue that should have been caught by an environment representative of lightweight Linux CI.
- **Suggestions:**
  - add a CI job without Tk and assert GUI tests are skipped cleanly;
  - add at least one smoke test or module-level test proving the skip path works when Tk cannot be imported.

### C) Development docs do not mention the Linux Tk dependency

- **Area:** `README.md`
- **Severity:** Minor
- **Observation:** The development setup instructions only say:
  ```bash
  python -m pip install -e .
  ```
  but do not mention that Linux users may also need the OS-level Tk package (for example `python3-tk`, depending on distro).
- **Why it matters:** Following the documented setup is currently insufficient to run the GUI or collect GUI tests in minimal Linux environments.
- **Suggestions:**
  - add a short Linux note near the development install/GUI launch sections;
  - mention that Tk is a system dependency, not a pip dependency.

## Overall assessment

- The new cleanup and palette changes appear functionally healthy in the non-GUI paths tested here.
- The main actionable defect is **test portability**: the default suite currently fails on Tk-less Linux runners before executing the majority of tests.
- The strongest follow-up after fixing the collection issue is to address the new gap-fill path's scaling behavior for larger images.
